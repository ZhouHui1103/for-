# ADHD业务指标体系 (v2.2 - 格式统一版)

---

## 1. 概述

本文件旨在将《指标体系规范》应用于“ADHD督导业务”，为该业务线定义一套**完备、可落地、可追溯**的核心指标。所有指标的定义均严格遵循规范，并确保最终可回溯至L2层业务对象底表。每个指标卡片都经过了端到端溯源验证。

---

## 2. 核心财务指标 (Financial Metrics)

#### **指标卡片 FIN-01: 流水 (GMV)**
- **关联对象**: `订单信息`
- **业务定义**: 在指定统计周期内，所有已支付订单的总金额。
- **计算逻辑**: `SUM(订单金额)` WHERE `订单状态` IN ('交付中', '交付结束')
- **统计粒度**: `日` / `周` / `月`
- **分析维度**: `产品清单`、`订单类型` (新客/复购)、`来源渠道`
- **数据来源**: `订单信息表`
- **[溯源验证]**:
  - `订单信息`对象: 已在L2定义。
  - `订单金额`: 可通过`订单信息`关联`产品清单`的`价格`字段获取。
  - `订单状态`: 已在`订单信息`对象中定义。
  - **结论: 可追溯。**

#### **指标卡片 FIN-02: 人力成本**
- **关联对象**: `人员信息`, `薪酬卡片` (源自《ADHD薪酬》)
- **业务定义**: 在指定统计周期内，支付给所有角色的总薪酬，包括基础、绩效和激励。
- **计算逻辑**: `SUM(角色最终薪酬)`，其中`最终薪酬`需根据《ADHD薪酬》文档中的XC系列规则进行计算。
- **统计粒度**: `月`
- **分析维度**: `角色类型` (督导师/培训师/支撑运营)
- **数据来源**: `薪酬计算表` (需新建), `业务跟踪表`
- **[溯源验证]**:
  - 薪酬计算依赖的`服务数量`、`续费时间长度`、`好评订单数量`均可通过`服务进程`和`订单信息`计算得出。
  - `绩效系数`依赖JX系列绩效卡片，其评估标准（如好评率）同样源自L2对象。
  - **结论: 可追溯，但强依赖《ADHD薪酬》文档的计算逻辑。**

#### **指标卡片 FIN-03: 业务利润/利润率**
- **关联对象**: `订单信息`, `薪酬卡片`
- **业务定义**: 周期内的总流水减去总人力成本。利润率 = 利润 / 流水。
- **计算逻辑**: `(流水 - 人力成本) / 流水`
- **统计粒度**: `月` / `季`
- **分析维度**: `产品线`
- **数据来源**: `订单信息表`, `薪酬计算表`
- **[溯源验证]**:
  - 依赖于`FIN-01 流水`和`FIN-02 人力成本`的计算结果。
  - **结论: 可追溯。**

## 3. 核心结果指标 (Outcome Metrics)

#### **指标卡片 OUT-01: 客户增长**
- **指标名称**: **周新增付费客户数 (增量)**
- **关联对象**: `客户信息`, `订单信息`
- **业务定义**: 在本周内，首次成功支付订单的独立客户数量。
- **计算逻辑**: `COUNT(DISTINCT 客户ID)` WHERE `订单类型` = '初次' AND `支付时间` 在本周内
- **统计粒度**: `周`
- **分析维度**: `来源渠道`、`需求属性` (学习型/工作型)
- **数据来源**: `订单信息表`, `客户信息表`
- **[溯源验证]**:
  - `订单类型`中的'初次'在L2 `订单信息`对象中已定义。
  - **[数据依赖]**: `订单信息表`需包含精确的`支付时间`字段，以确认订单的发生时间。
  - **结论: 可追溯，存在数据依赖。**

#### **指标卡片 OUT-02: 客户留存**
- **指标名称**: **客户复购率**
- **关联对象**: `服务进程`, `订单信息`
- **业务定义**: 周期内所有结束服务的客户中，后续产生了新“复购”订单的客户占比。
- **计算逻辑**: `COUNT(DISTINCT 客户ID WHERE 存在复购订单) / COUNT(DISTINCT 客户ID WHERE 服务状态 = '已结束')`
- **统计粒度**: `月`
- **分析维度**: `产品清单`、`督导小队`、`督导师`
- **数据来源**: `服务进程跟踪表`, `订单信息表`
- **[溯源验证]**:
  - `服务状态='已结束'`在`服务进程`对象中已定义。
  - “存在复购订单”可通过`订单信息`的`订单类型='复购'`来判断。
  - **结论: 可追溯。**

#### **指标卡片 OUT-03: 客户满意度**
- **指标名称**: **客户满意度 (CSAT)**
- **关联对象**: `客户主观反馈`
- **业务定义**: 周期内，所有客户反馈评分的算术平均值。
- **计算逻辑**: `AVG(评分)`
- **统计粒度**: `周` / `月`
- **分析维度**: `产品清单`、`督导小队`、`督导师`、`反馈类型`
- **数据来源**: `客户主观反馈表`
- **[溯源验证]**:
  - `客户主观反馈`对象及其`评分`字段已在L2定义。
  - **结论: 可追溯。**

#### **指标卡片 OUT-04: 服务口碑**
- **指标名称**: **服务好评率**
- **关联对象**: `服务进程`
- **业务定义**: 周期内所有结束的服务中，被标记为“好评”的服务占比。
- **计算逻辑**: `COUNT(服务进程ID WHERE 是否好评 = true) / COUNT(服务进程ID WHERE 状态 = '已结束')`
- **统计粒度**: `月`
- **分析维度**: `产品清单`、`督导小队`、`督导师`
- **数据来源**: `服务进程跟踪表`
- **[溯源验证]**:
  - L2 `服务进程`对象的`数据字段`中已包含`是否好评`。
  - **结论: 可追溯。**

#### **指标卡片 OUT-05: 服务质量**
- **指标名称**: **服务交付质量分**
- **关联对象**: `服务进程质量评估`
- **业务定义**: 周期内，所有由培训师完成的质量评估的算术平均分。
- **计算逻辑**: `AVG(评估得分)`
- **统计粒度**: `周` / `月`
- **分析维度**: `培训师`、`督导小队`、`督导师`
- **数据来源**: `服务进程质量评估表`
- **[溯源验证]**:
  - `服务进程质量评估`对象及其`评估得分`字段已在L2定义。
  - **结论: 可追溯。**

#### **指标卡片 OUT-06: 团队健康度**
- **指标名称**: **人员流失率**
- **关联对象**: `人员信息`
- **业务定义**: 周期内，离职员工数 / (期初员工数 + 期末员工数) / 2。
- **计算逻辑**: `COUNT(人员ID WHERE 状态='离职' AND 离职时间在周期内) / AVG(期初总人数, 期末总人数)`
- **统计粒度**: `季` / `年`
- **分析维度**: `角色类型` (督导师/培训师)
- **数据来源**: `人员信息表`
- **[溯源验证]**:
  - `人员信息`对象的`状态`包含'离职'。
  - **[数据依赖]**: `人员信息表`需增加`离职时间`字段，以支持按周期统计。
  - **结论: 可追溯，存在数据依赖。**

## 4. 核心过程指标 (Process Metrics)

#### **指标卡片 PRO-01: 售前效率**
- **指标名称**: **订单分配平均时长**
- **关联对象**: `订单信息`, `服务进程`
- **业务定义**: 从客户成功下单到运营完成服务进程创建和分配的平均耗时。
- **计算逻辑**: `AVG(服务进程创建时间 - 订单创建时间)`
- **统计粒度**: `日` / `周`
- **分析维度**: `支撑运营人员`
- **数据来源**: `订单信息表`, `服务进程跟踪表`
- **[溯源验证]**:
  - **[数据依赖]**: 底层数据表需记录精确的`订单创建时间`和`服务进程创建时间`。L2文档中虽未显式定义，但流程的启动意味着此时间戳必须存在。
  - **结论: 可追溯，强数据依赖。**

#### **指标卡片 PRO-02: 服务启动效率**
- **指标名称**: **服务启动平均时长**
- **关联对象**: `服务进程`
- **业务定义**: 从服务进程被创建，到Onboarding完成、服务状态变为“进程中”的平均耗时。
- **计算逻辑**: `AVG(服务状态变为'进程中'时间 - 服务进程创建时间)`
- **统计粒度**: `日` / `周`
- **分析维度**: `支撑运营人员`
- **数据来源**: `服务进程跟踪表`
- **[溯源验证]**:
  - **[数据依赖]**: `服务进程跟踪表`需记录每一个状态（如`待开启`、`进程中`）的进入时间戳。
  - **结论: 可追溯，强数据依赖。**

#### **指标卡片 PRO-03: 日常服务效率**
- **指标名称**: **待办任务平均完成时长**
- **关联对象**: `最小待办单元`
- **业务定义**: 从待办任务被创建，到被标记为“已完成”的平均耗时。
- **计算逻辑**: `AVG(完成时间 - 提出时间)`
- **统计粒度**: `日` / `周`
- **分析维度**: `督导师`、`待办类型`
- **数据来源**: `最小待办单元表`
- **[溯源验证]**:
  - `最小待办单元`对象及其`完成时间`和`提出时间`字段已在L2定义。
  - **结论: 可追溯。**

#### **指标卡片 PRO-04: 日常服务质量**
- **指标名称**: **每日A级计划占比**
- **关联对象**: `每日计划进度`
- **业务定义**: 周期内，所有被记录的每日计划中，评估等级为“A”的占比。
- **计算逻辑**: `COUNT(记录ID WHERE 评估等级 = 'A') / COUNT(所有记录ID)`
- **统计粒度**: `周`
- **分析维度**: `督导师`、`客户`
- **数据来源**: `每日计划进度表`
- **[溯源验证]**:
  - `每日计划进度`对象及其`评估等级`字段已在L2定义。
  - **结论: 可追溯。**

#### **指标卡片 PRO-05: 资源利用**
- **指标名称**: **督导小队容量利用率**
- **关联对象**: `督导小队`
- **业务定义**: 某个时间点，所有督导小队已负责的服务数量占总容量的比例。
- **计算逻辑**: `SUM(已负责数量) / SUM(总容量)`
- **统计粒度**: `日` (快照)
- **分析维度**: `服务时段` (早班/晚班)
- **数据来源**: `督导小队表`
- **[溯源验证]**:
  - `督导小队`对象及其`已负责数量`和`总容量`字段已在L2定义。
  - **结论: 可追溯。**
