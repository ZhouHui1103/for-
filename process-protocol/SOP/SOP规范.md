这套规范旨在确保SOP的清晰性、可执行性和可管理性，是编写高质量SOP时应遵循的核心原则。

## 一、文档结构说明

### **组成逻辑**
SOP文档采用**对象驱动的结构设计**，按照以下逻辑组织：

1. **文档元信息** - 版本日志、流程概述、触发条件、角色关系、场域概览
2. **对象操作块** - 每个对象对应一个独立的大标题区块，采用L4-L5-L6层级结构
3. **协调支撑块** - 沟通协调、指南说明、质量检查

### **层级结构规范**
- **L3 (流程)**: 指整个SOP文档，描述一个完整的端到端业务流程，或者一组逻辑上相关的业务活动的集合。
- **L4 (阶段/对象/场景)**: SOP的二级标题（`##`）。其含义根据SOP的类型而定：
    - **对于单一对象驱动的流程**: L4代表该对象的**一个处理阶段**。
    - **对于合并后的宏观流程**: L4代表一个**具体的子流程或业务场景**。
- **L5 (活动)**: L4下的编号列表（`1.` `2.`），表示一个具体的"活动"，标题必须是**动词短语**
- **L6 (动作)**: L5下的字母列表（`a.` `b.`），是完成L5活动的**具体、连续的动作步骤**。**每个L6动作都必须明确指出操作的具体`对象`和`字段`。**

### **对象组织原则**
- **按业务重要性排序**: 主要状态变量对象优先
- **按时间顺序排列**: 遵循对象在流程中的处理顺序
- **按依赖关系组织**: 被依赖的对象先于依赖它的对象

### **文档元信息要求**
SOP正文开始前，必须包含以下元信息：
- **版本日志**: 记录文档的变更历史
- **流程概述**: 简要说明该SOP的目标、解决的问题及其在整体业务中的位置
- **触发条件**: 描述启动此SOP需要满足的前置条件，**应具体到某个`场域`中的某个`对象`的状态**
- **角色关系**: 列出并说明此流程中涉及的所有角色及其核心职责
- **场域概览**: 
  - **数据场域**: 列出执行此流程所涉及的所有**数据存取**平台
  - **沟通场域**: 列出执行此流程所涉及的所有**沟通协作**平台
  - **文档场域**: 列出相关的文档资源（如适用）

### **使用场景**
- **流程执行指导**: 作为一线执行人员的操作手册
- **培训教材**: 新人学习业务流程的标准材料
- **质量检查**: 流程审核和问题排查的检查清单
- **流程优化**: 识别瓶颈和改进点的分析基础

### **项目关系**
- **设计阶段**: 基于L2元数据定义，在流程设计阶段创建
- **实施阶段**: 在项目实施过程中作为执行依据
- **运行阶段**: 作为日常操作的标准规范
- **优化阶段**: 根据实际执行情况进行迭代更新

### **角色定义**
- **维护人**: `流程设计师` - 负责SOP的日常更新和维护
- **负责人**: `业务负责人` - 对SOP的准确性和实用性负责
- **审核人**: `质量管理员` - 负责审核SOP的规范性和完整性
- **使用人**: `一线执行人员` - SOP的实际使用者和反馈提供者

## 二、填写指南

### **基础填写规则**

#### **标题和版本管理**
- **文档标题**: 格式为 `L3 - [流程编号] [流程名称]`
- **流程编号**: 按照 `模块.子模块.序号` 的格式，如 `2.1.3`
- **版本日志**: 每次修改必须更新版本号和修改记录

#### **流程基础信息填写**
- **流程概述**: 2-3句话说明流程目标和在整体业务中的作用
- **触发条件**: 必须具体到数据场域中对象的状态，避免模糊表述
- **角色关系**: 明确主导和协同关系，避免责任不清
- **场域概览**: 列出所有涉及的场域，确保信息传递完整

#### **对象块填写原则**
每个对象块必须包含：
1. **对象基础信息** - 对象类型、关键属性
2. **具体操作内容** - 字段操作、状态变更、创建规则等
3. **执行步骤** - 按L5活动级别详细描述操作步骤

#### **内容要求**
- **动作导向**: SOP正文只包含可执行、无歧义的动作。所有解释、原理、判断标准、案例等"思考型"内容，全部放入文末的**`指南说明`**部分
- **闭环操作与数据精准**: 每个L5活动都应形成"输入-处理-输出"的闭环。L6动作需明确：
  - **精准输入**: 必须明确从哪个**`数据场域`**的**具体位置**读取数据
  - **处理**: **执行**什么具体操作。为提高可读性，**建议将核心动词加粗**，如`**查看**`、`**新建**`、`**更新**`
  - **精准输出**: 必须明确向哪个**`数据场域`**的**具体位置**写入数据，且**必须明确到所操作的具体`字段`名称，并清晰说明其操作逻辑**（如：设为固定值、从另一字段复制、根据...计算、原子化增减等）。
- **核心处理留白**: 对于需要人工执行、决策的复杂业务动作，应**使用标准格式进行留白**
  - **留白格式**: `[L6动作留白 - *请补充[具体需要做什么]的细节*]`
  - **标准示例**: `角色`: `[L6动作留白 - *请补充评估该任务优先级的具体判断标准与决策过程*]`

#### **协作规范**
- **角色明确**: 每个L4阶段和L6动作都必须有明确的执行角色
- **交接清晰**: 当涉及角色交接时，必须有一个明确的L6动作，规定A角色在哪个`沟通场域`，使用什么"模板/格式"，通知B角色

#### **格式要求**
- **标题规范**: L4使用二级标题（`##`），L5使用数字列表，L6使用字母列表。**严禁出现三级标题或更深层级**
- **引用辅助**: 需要参考的话术、模板或格式要求，使用Markdown的**引用块**（`> ...`）附在对应的L6动作下方

### **状态关联填写**

#### **状态变量对象**
- **状态流转记录**: 每个状态变更都要有明确的前后状态标注
- **状态变更条件**: 必须明确触发状态变更的具体条件
- **字段更新规则**: 
  - 状态变更时的必更新字段
  - 依据状态的条件更新字段
  - 状态相关的计算字段更新

#### **任务属性对象**
- **任务状态同步**: 任务对象的状态变更要与原始对象状态保持同步
- **责任人变更**: 不同任务状态下责任人的调整规则
- **时间节点管理**: 
  - 任务创建时间与原始对象的关系
  - 排期时间的设定规则
  - 验收时间的记录要求

#### **督导属性对象**
- **督导时机调整**: 根据被督导对象的状态确定督导频率
- **督导内容变化**: 不同状态下督导检查的重点内容
- **督导结果处理**: 督导发现问题时的处理和跟踪机制

### **跨对象关联填写**
- **关联字段一致性**: 确保关联字段在所有相关对象中保持一致
- **联动更新规则**: 一个对象变更时，相关对象的自动更新机制
- **数据传递路径**: 明确信息在不同对象间的传递顺序和方式

### **时间节点要求**
- **创建时机**: SOP在L2元数据定义完成后立即创建
- **更新频率**: 
  - 对象定义变更时必须同步更新
  - 实际执行中发现问题时及时更新
  - 定期回顾（建议每季度）
- **版本控制**: 重大变更需要版本升级，小修改可以在当前版本基础上迭代

## 三、检查标准

### **完整性检查**

#### **全量检查标准**
- [ ] **流程覆盖度**: 是否涵盖了L2元数据中定义的所有相关对象？
- [ ] **状态完整性**: 每个状态变量对象的所有状态是否都有对应的处理步骤？
- [ ] **角色完整性**: L2中定义的所有角色关系是否在SOP中都有体现？
- [ ] **场域完整性**: 所有相关的数据场域、沟通场域是否都被正确引用？
- [ ] **字段覆盖度**: 对象的关键字段操作是否都有明确说明？

#### **阶段性检查标准**

**设计阶段检查**:
- [ ] 对象定义与L2元数据的一致性
- [ ] 流程逻辑的合理性和可执行性
- [ ] 角色权责的明确性和可操作性

**实施阶段检查**:
- [ ] 操作步骤的详细程度和准确性
- [ ] 异常情况的处理机制完备性
- [ ] 质量检查项的可执行性

**运行阶段检查**:
- [ ] 实际执行效果与设计预期的匹配度
- [ ] 用户反馈问题的及时修正
- [ ] 流程优化需求的识别和处理

### **准确性检查**

#### **数据一致性**
- **对象状态**: SOP中描述的状态流转与L2定义的状态路径一致
- **字段操作**: 字段更新规则与对象定义中的字段属性一致
- **角色权限**: 角色操作权限与L2中定义的角色关系一致
- **计算逻辑**: 涉及计算字段的操作与公式定义一致

#### **逻辑一致性**
- **前后逻辑**: 前置条件与后续操作的逻辑关系正确
- **分支处理**: 条件分支的覆盖性和互斥性
- **异常处理**: 异常情况的处理逻辑完整且合理
- **回滚机制**: 操作失败时的回滚和恢复逻辑

### **时效性检查**

#### **更新响应性**
- **对象变更响应**: L2元数据变更后1周内完成SOP更新
- **问题反馈响应**: 执行中发现问题后3天内完成分析和修正
- **定期回顾**: 每季度进行一次全面回顾和更新评估

#### **版本管理**
- **版本记录**: 每次变更都有完整的版本记录和变更说明
- **版本控制**: 重大变更(影响>30%内容)需要版本号升级
- **历史保留**: 保留至少最近3个版本的历史记录

### **质量标准**

#### **格式规范**
- **标题层级**: 严格按照H2-H4的层级结构组织内容
- **对象标识**: 对象名称使用代码块格式高亮显示
- **操作动作**: 关键操作动作使用加粗格式强调
- **检查清单**: 质量检查项使用checkbox格式

#### **内容深度要求**
- **L5活动**: 每个执行步骤都要细化到L5活动级别
- **具体操作**: 每个操作都要有明确的执行主体、操作对象、**操作字段**和**具体的操作逻辑/内容**
- **判断标准**: 涉及判断的环节都要有明确的判断标准
- **异常处理**: 常见异常情况都要有对应的处理方案

#### **监测点设置规范**
SOP文档末尾应包含一组关键的检查点，用于评估SOP执行质量：
- **结果导向**: 每个监测点都应是一个**可验证的产出物**，而不是过程动作
- **明确具体**: 每个监测点都应**明确指向某个`数据场域`中的某条`记录`的某个`字段`**是否符合预期状态
- **L1场景闭环**: 监测点的设置必须能**完全覆盖**并验证对应`L1业务场景`中定义的所有`验收标准`
- **标准示例**:
  - `[ ]` `服务进程跟踪表`中是否已成功创建`服务进程`记录，且`状态`为`待开启`？
  - `[ ]` 原`订单信息表`中的记录，`状态`是否已正确更新为`待交付`？
  - `[ ]` `督导小队群`中，新服务分配通知是否已按模板格式正确发布并@相关人员？

## 四、质量控制流程

### **创建阶段质量控制**
1. **自检**: 创建者按照检查标准进行自查
2. **同行评议**: 同级流程设计师进行交叉检查
3. **专家审核**: 业务专家对内容准确性进行审核
4. **用户测试**: 目标用户进行可用性测试

### **更新阶段质量控制**
1. **变更评估**: 评估变更影响范围和重要程度
2. **增量检查**: 重点检查变更部分的准确性和一致性
3. **回归测试**: 确保变更不影响其他部分的正确性
4. **发布确认**: 正式发布前的最终确认

### **持续改进机制**
- **用户反馈收集**: 建立用户反馈收集和处理机制
- **执行效果监控**: 定期监控SOP执行效果和问题
- **最佳实践提炼**: 总结和推广优秀的SOP设计经验
- **工具支持优化**: 持续优化SOP创建和维护的工具支持
