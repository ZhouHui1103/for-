## 核心任务

**实现一个能够批量生产宣传素材的自动化脚本。**

具体场景：假设你需要为一系列线上活动制作宣传图，每张图的背景、版式都相同，只有文字内容不同。你需要开发一个脚本，根据提供的文本，自动生成符合要求的图片素材。

**重要提示**：本次任务需要生成的素材样式，请参考此链接： `[此处放置参考样式的笔记链接]`。在开始前，请仔细观察并总结参考图片中的核心排版要点（如字体、字号、间距、颜色等），这会是 TDD 流程中重要的对比基准。

## 详细要求

- **输入**: 脚本接受一个或多个文字文本作为参数。

## 资源与指南

### 1. Claude Code 安装与配置

1.  **全局安装**:
    在终端中执行以下命令。你可以直接在 Cursor 的终端中进行，如果遇到权限问题，它会引导你解决。
    ```bash
    npm install -g @anthropic-ai/claude-code
    ```
2.  **配置 Kimi 模型**:
    本次任务要求使用硅基流动（Moonshot AI）的 Kimi 模型。`Claude Code` 本身不直接支持第三方模型，但我们可以通过配置环境变量，将其API请求代理到 Kimi 的服务终端，这能更好地考察你的动手能力。

    - **获取API Key**: 首先，你需要拥有一个有效的 Moonshot AI API Key。
    - **设置环境变量**: 在运行 `Claude Code` 之前，在你的终端中设置以下环境变量：

      ```bash
      # 1. 将 Claude Code 的 API 端点指向 Moonshot AI 的服务地址
      export ANTHROPIC_API_URL="https://api.moonshot.cn/v1"

      # 2. 设置你的 Moonshot AI API Key
      export ANTHROPIC_API_KEY="YOUR_MOONSHOT_API_KEY"
      ```
    - **指定模型**: 在与 `Claude Code` 对话时，你可能需要通过特定指令或在初始设置中指定使用的具体模型，例如 `moonshot-v1-8k` 或 `moonshot-v1-32k`。请参考 Moonshot AI 的官方文档进行确认。

    **注意**：这是一种探索性的配置方式，如果遇到问题，请记录你的尝试过程和解决方案。

3.  **开启 YOLO 模式 (推荐)**:
    YOLO (You Only Live Once) 模式将给予 `Claude Code` 完全的自主执行权限，免去手动确认的步骤，提高效率。
    ```bash
    alias cc-yolo="CLAUBBIT=1 CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1 ENABLE_BACKGROUND_TASKS=1 claude --dangerously-skip-permissions"
    ```
    后续在终端中使用 `cc-yolo` 命令即可进入此模式。普通模式则继续使用 `claude`。

### 2. Playwright MCP 配置

1.  **添加 MCP**:
    直接在 `Claude Code` 中运行以下命令，即可添加 Playwright MCP。
    ```bash
    /mcp add playwright npx @playwright/mcp@latest
    ```
2.  **使用**:
    添加后，在 `Claude Code` 输入 `/mcp`，初次使用会显示「Connecting」状态，稍等片刻即可连接成功并使用。

### 3. “自动迭代”与测试驱动开发（TDD）哲学

我们提到的“自动迭代”，本质上是测试驱动开发（TDD）思想在 AI 协同工作流中的一种应用。TDD 的核心循环是“红-绿-重构”，可以这样理解：

-   **红（Red）**: 先失败。针对你的目标，首先创建一个自动化测试，用于验证最终产出的图片是否符合预期。例如，你可以先准备一张“目标图片”，然后让测试脚本去对比 `Claude Code` 生成的图片和这张目标图片。由于代码还没写，这个测试一开始必然是失败的。
-   **绿（Green）**：求成功。接下来，驱动 `Claude Code` 编写最少的代码，让上面的测试通过。目标是尽快让生成的图片与“目标图片”在像素层面完全一致，使测试变“绿”。
-   **重构（Refactor）**：做优化。在测试通过的前提下，再回过头来优化和重构你的代码、脚本或指令（Prompt），使其更健壮、更优雅、更高效，同时确保测试依然是“绿”的。

**给你的提示：**

在这个任务中，你可以指示 `Claude Code` 遵循这个流程：
1.  **定义目标**：首先给 `Claude Code`一张完美的“参考截图”。
2.  **创建测试**：可以让 `Claude Code` 编写一个简单的脚本或使用工具来对比两张图片（它生成的 vs 你给的参考）的差异，甚至也可以直接让 `Claude Code` 自行对比，因为大模型具备对比图片的能力。
3.  **迭代开发**：命令它不断修改生成图片的脚本，直到图片对比测试通过为止。
4.  **最终交付**：交付的脚本应该是这个迭代过程最终的、成功的版本。

这种方式不仅能保证产出质量，更能体现出一种严谨、闭环的工程化开发思维。

### 4. 利用文档实现“记忆”与“日志”

`Claude Code` 本身是无状态的，它不会“记住”上一次复杂任务的上下文。为了解决这个问题，我们可以巧妙地利用本地文件作为它的“外部记忆体”。

**核心思路**：将需要长期遵循的指令、中间状态或历史记录写入文档，然后在需要时，让 `Claude Code` 读取这些文档来“回忆”起上下文或接下来的任务。

**具体技巧：**

1.  **制定行动指南 (`cc-runner.md`)**：
    你可以创建一个 `Markdown` 文件，在里面用自然语言或伪代码写下清晰、分步骤的行动计划。然后，你可以让 `Claude Code` 读取这个文件，并像执行任务清单一样去完成工作。
    *   **示例指令**：“`Claude Code`，请读取并执行 `cc-runner.md` 中的所有步骤。”
    *   **好处**：这使得复杂的任务流程变得清晰、可重复和易于调试。

2.  **记录研发日志 (`dev_log.md`)**：
    在交付物中，我们要求你提交一份研发日志。你可以通过给 `Claude Code` 下达一个初始指令来实现这一点，让它在后续的每一步操作后，都将自己的思考、执行的命令、遇到的问题和解决方案追加（append）到 `dev_log.md` 文件中。
    *   **示例指令**：“`Claude Code`，在接下来的任务中，请将你的每一个步骤和思考过程都记录到 `dev_log.md` 文件里。”
    *   **好处**：这不仅能满足交付要求，更能帮助你追踪和复盘整个开发过程。

3.  **默认规则 (`claude.md`)**：`claude.md` 是一个关键文档，默认情况下每次任务开始时都会被调用。它包含了 `Claude Code` 的核心行为准则和全局指令，确保其操作符合预期。
    *   **自动加载**：在每次任务启动时，`Claude Code` 会自动读取 `claude.md`，以确保其行为与项目的基本要求一致。
    *   **全局指令**：该文档中记录的指令适用于所有项目，提供了一致的操作基础。

4.  **沉淀抽象规则 (`rules.md`)**：`rules.md` 用于记录项目中特定的抽象规则和设计模式。它包含了在对话中沉淀下来的关键原则和常用指令。
    *   **动态沉淀**：在对话过程中，可以让 `Claude Code` 自行总结并记录到 `rules.md` 中。
    *   **领域知识库**：为特定领域（如UI规范、API调用格式等）建立专门的规则文档。
    *   **按需引用**：在开始新任务时，让 `Claude Code` 先读取相关的规则文档，确保其行为符合预设的规范。
    这是一种更高级的用法，其本质是**记录并复用抽象规则**。你可以让 `Claude Code` 把对话中沉淀下来的关键原则、设计模式或常用指令，保存到一个或多个专门的规则文档中。
    *   **动态沉淀**：在对话过程中，你可以让 `Claude Code` 自行总结。例如：“`Claude Code`，根据我们刚才的讨论，请将关于图片生成的几条核心规则，沉淀并追加到 `image_rules.md` 文件中。”
    *   **领域知识库**：你也可以为特定领域（如UI规范、API调用格式等）建立专门的规则文档。
    *   **按需引用**：在开始新任务时，让 `Claude Code` 先读取相关的规则文档，确保它的行为符合预设的规范。例如：“`Claude Code`，在开始编码前，请先学习 `api_rules.md` 中的所有规则。”
    *   **命名建议**：通常，项目相关的具体规则可放入 `rules.md`，而那些针对 `Claude Code` 行为本身的、更通用的全局指令，可以考虑放在 `claude.md` 文件中，以便跨项目复用。

### 5. 善用官方“计划模式”（Plan Mode）

为了提升处理复杂任务的准确性和安全性，`Claude Code` 官方提供了“计划模式”（Plan Mode）。其核心思想是：**先让 `Claude Code` 自动分析并制定详细计划，待你确认后，再授权执行**。

**如何启用计划模式？**
*   **会话中切换**: 在交互式会话里，可以随时通过快捷键 `Shift + Tab` 切换进入计划模式。

**计划模式的工作流程:**

1.  **任务分析**: 你下达指令后，`Claude Code` 首先会分析你的需求，探索相关代码，识别需要修改的文件和潜在影响。
2.  **计划制定**: 基于分析，它会制定一份详细的行动计划，其中包含具体步骤、要修改的文件列表和预期的更改内容。
3.  **用户确认**: `Claude Code` 会将这份完整的计划展示给你。此时，你可以审查、提出修改建议，或者直接批准该计划。这是执行前一个关键的“刹车点”。
4.  **授权执行**: 一旦你确认计划，`Claude Code` 才会开始按照计划执行实际的编码和操作。

**为什么要使用计划模式？**

*   **提升安全性与可预测性**: 在执行任何破坏性操作前，让你能清晰预知将要发生的所有更改，避免意外的代码损坏。
*   **保证任务完整性**: 对于涉及多文件、多步骤的复杂任务，计划模式能确保所有环节都被考虑到，不会遗漏关键步骤。
*   **推荐使用场景**: 当你首次接触一个新代码库、准备进行大规模重构，或任何不确定更改影响范围的场景时，强烈建议使用计划模式。

通过善用这一官方功能，你可以更安全、高效地驾驭 `Claude Code`，确保最终的交付质量。

### 6. Vibe Coding 核心原则

Vibe Coding 是一种与 AI 深度协同的工作哲学。为了帮助你更好地实践，我们总结了以下三个核心原则：

1.  **信任AI的自主性，专注于高层抽象**:
    尽可能将具体工作授权给 `Claude Code`，包括梳理需求、分析现状、对比差异，甚至主动提出改进建议。你的角色应更多地是提供最核心、最抽象的思路，并在关键节点（如审美、架构决策）进行引导和修正，而不是事必躬亲地指导每一步。

2.  **善用外部记忆，保持上下文专注**:
    AI 的短期记忆是有限的。要学会通过文档（如 `rules.md`, `dev_log.md`）来沉淀关键经验和过程。在需要 `Claude Code` 专注解决某个局部问题时，可以果断地清空（Clear）当前对话上下文，然后让它通过读取文档来获取所需的核心知识。这就像为它配备了一个可随时查阅的、永不遗忘的“第二大脑”。

3.  **主导AI驱动的快速迭代**:
    相信 `Claude Code` 在微观层面的迭代速度和能力超越人类。你的核心任务是**定义好目标**（例如一张完美的参考图片），并**主持好“迭代-测试-反馈”这个循环**。让 `Claude Code` 自主地、快速地朝目标迭代，识别问题，并列出改进项，然后继续迭代。你只需控制好这个反馈循环的节奏和最终目标，让 AI 成为自我优化的主角。
