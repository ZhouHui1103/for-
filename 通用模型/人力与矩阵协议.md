# 人力与矩阵协议：形式化定义 V1.1

## 1. 概述

本文档旨在严格遵循《Modulon 协议白皮书》与《Modulon 协议黄皮书》中定义的核心概念，对“人力协议”与“矩阵协议”进行形式化的、无歧义的定义。

- **人力协议 (Human Protocol)**：并非一个单一的协议 `Ρ`，而是与人力资源相关的**模块镜像 (`Φ_H`)** 和**节点 (`Ν_H`)** 的集合。它为人类的技能、角色和任务提供了标准化的“能力蓝图”。
- **矩阵协议 (Matrix Protocol)**：是一个具体的、复合型的**协议 (`Ρ_M`)**，它通过编排包括人力模块在内的多个模块镜像，来定义一个完整的业务流程或组织结构。

---

## 2. 人力协议 (Human Protocol) 的形式化定义

人力协议是 Modulon 协议在人力资源领域的具体应用，其核心是定义构成“人力”的原子化能力单元。

### 2.1 人力模块镜像 (`Φ_H`)

人力模块镜像是对一个标准化人力服务单元的抽象规范。它遵循黄皮书中 `Φ` 的数据结构。

**`Φ_H` ⊂ `S_Φ`** (人力模块镜像的集合是总模块镜像集合的子集)

**示例：定义一个“增长人力招募”模块镜像 `Φ_H_GrowthRecruitment`**

```
Φ_H_GrowthRecruitment := {
    id: "d5a2f8b1-c3e2-4f6a-8a99-4e1b2c3d4e5f",
    spec_version: "1.0.0",
    ontology_uri: "/ontology/human_resource/recruitment",
    inputs: {
        TypedParam("招募计划", "RecruitmentPlanV1"),
        TypedParam("招募渠道列表", "List[Channel]")
    },
    outputs: {
        TypedParam("合格候选人线索", "List[CandidateLead]")
    },
    eval_logic: Hash("logic_hash_recruitment_quality"), // 指向评估招募质量（如入职率、留存率）的逻辑
    pricing_model: Hash("price_hash_per_candidate"),   // 指向按成功招募人数计价的模型
    metadata: {
        "name": "增长人力招募模块",
        "description": "负责根据招募计划，通过指定渠道获取并筛选候选人线索。"
    }
}
```

### 2.2 人力节点 (`Ν_H`)

人力节点是声明其有能力实现一个或多个 `Φ_H` 的、可提供具体人力服务的网络实体。它可以是一个HR个人、一个招聘团队，甚至是一个自动化的招聘Bot。

**`Ν_H` ⊂ `S_Ν`**

**示例：定义一个实现“增长人力招募”的节点 `Ν_H_TeamAlpha`**

```
Ν_H_TeamAlpha := {
    id: "0xAddressOfTeamAlpha",
    owner: "0xAddressOfTeamLead",
    mirrors: { "d5a2f8b1-c3e2-4f6a-8a99-4e1b2c3d4e5f" }, // 实现了上述 Φ_H_GrowthRecruitment
    state: "Active",
    rating: { "d5a2f8b1-c3e2-4f6a-8a99-4e1b2c3d4e5f": 4.85 }, // 在该模块下的历史评级
    endpoint: "api.team-alpha.com/hr/recruit",
    stake: Balance(1000)
}
```

**结论**：人力协议的本质，是**所有 `Φ_H` 的集合**，以及由所有 `Ν_H` 构成的、围绕这些能力蓝图展开的服务供给市场。它本身不定义复杂的业务流程，只提供标准化的“人力积木”。

---

## 3. 矩阵协议 (Matrix Protocol) 的形式化定义

矩阵协议是一个上层的、复合型的业务流程总蓝图，它通过编排（Orchestration）包括人力协议在内的各种模块，来定义一个完整的业务矩阵。

### 3.1 矩阵协议的定义 (`Ρ_M`)

矩阵协议严格遵循黄皮书中 `Ρ` 的数据结构，其核心在于 `orchestrated_mirrors` 字段，它显式地声明了所包含的子模块，其中必然会有人力模块 `Φ_H`。

**`Ρ_M` ⊂ `S_Ρ`**

**示例：定义一个“策划中心业务”矩阵协议 `Ρ_M_PlanningCenter`**

```
Ρ_M_PlanningCenter := {
    id: "c8b7e1a0-d3e4-5f6a-7b8c-9d0e1f2a3b4c",
    protocol_spec_version: "1.0.0",
    mirror_id: "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d", // 此协议对外也表现为一个高级模块 Φ_Service_Planning
    orchestrated_mirrors: {
        "uuid_planning_spec",       // 策划规格定义模块
        "uuid_planning_execution",  // 策划执行模块
        "uuid_planning_review",     // 策划审核模块
        "d5a2f8b1-c3e2-4f6a-8a99-4e1b2c3d4e5f"  // <--- 此处包含了人力协议中的 Φ_H_GrowthRecruitment
    },
    rules: Hash("rules_hash_planning_workflow"), // 指向一个状态机或工作流脚本，定义以上模块的协作顺序、数据流和触发条件
    metadata: {
        "name": "策划中心业务矩阵协议",
        "description": "定义了从策划、执行、审核到人力支持的完整策划业务流程。"
    }
}
```

### 3.2 关系阐述：矩阵协议如何“包含”人力协议

从上述形式化定义可以看出，“矩阵协议包含人力协议”这一命题，在技术上体现为：

**一个矩阵协议 `Ρ_M` 的 `orchestrated_mirrors` 集合中，必然包含一个或多个来自人力协议的模块镜像 `Φ_H`。**

人力协议提供了一系列标准化的、可被调用的“人力服务API”（即 `Φ_H` 集合），而矩阵协议则是这些API的“调用方”和“编排者”。它将人力服务与业务服务（如策划、制作）无缝地组合在一起，定义了一个权责清晰、流程自动化、价值可计算的宏观业务模式。

---

## 4. 总结

- **人力协议**：一系列 `Φ_H` 和 `Ν_H` 的集合，是人力能力的**标准化规范层**。
- **矩阵协议**：一个具体的 `Ρ_M` 实例，是调用并编排 `Φ_H` 及其他业务 `Φ` 的**业务流程层**。

这种分层、组合的范式，完全遵循 Modulon“万物皆模块”的核心理念，为构建复杂、高效、去中心化的智力协作组织提供了坚实的技术基础。
