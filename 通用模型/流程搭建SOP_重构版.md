# 标准化流程搭建SOP (v6)

本文档旨在提供一个以“业务对象”为核心，从顶层设计、系统推演到部署验证的标准化作业流程（SOP）。

## 1. 核心设计哲学与基本要素

我们的组织搭建和流程设计遵循一个核心哲学：**一切始于业务对象**。业务对象是业务的本质，它决定了我们需要什么样的流程、数据、文档，以及由谁（即“节点”）来执行。

整个系统由三个基本要素构成：
- **场域 (Domain)**: 业务发生的上下文或场景。
- **对象 (Object)**: 业务流程中操作和管理的核心实体，是所有设计的起点。
- **节点 (Node)**: **一个统一且可分层的概念**，指代流程中执行操作的单元。**高层级的节点**（如“XX业务部”）对某个核心对象的全生命周期负责，而**最小执行单元**则指具体的个体、一段代码或一个AI Agent。**所有后续的文档和表格设计，都必须体现节点的层级关系**。
- **流程 (Process)**: 驱动对象状态发生变化的一系列有序活动。

## 2. 系统化推导逻辑 (重构)

我们的流程设计强调逻辑性和可推演行。一部分核心概念需要人为进行顶层设计（草稿），而大部分产出物都可以基于这些顶层设计系统化地推导出来。

### 阶段一：人为顶层设计 (Foundational Input / 草稿)
此阶段的目标是快速输出核心业务框架的草稿，作为后续系统化推演的基础。
1.  **业务故事草稿 (`文档`)**: 从价值主张出发，描述业务要解决的核心问题和初步设想。
2.  **核心对象清单 (初步) (`文档`)**: 识别出业务故事中最核心的名词，作为初始的业务对象。
3.  **节点清单 (初步) (`文档`)**: 对业务单元进行分层，定义出不同层级的“节点”草稿。
4.  **场域清单 (初步) (`文档`)**: 识别业务发生的上下文或场景。
5.  **计费思路 (`文档`)**: 初步构思与节点价值挂钩的激励机制。
6.  **流程清单 (初步) (`文档`)**: 粗略列出驱动对象状态变化的核心流程。

### 阶段二：系统化推演 (Systematic Derivation)
基于草稿，系统化、结构化地推演出完整的业务和系统设计。

#### 步骤1：节点、财务与业务框架构建
此步骤的重点是正式确立业务的参与者、商业模式和核心叙事。
1.  **梳理分层节点清单 (`表格`+ `文档`)**: 基于草稿，正式、详细地定义分层的业务节点及其职责，产出《节点手册》。
2.  **撰写完整业务故事 (`文档`)**: 结合节点、对象和流程，撰写包含完整本体论信息的业务故事。
3.  **定义计费与考核标准 (`文档`+`表格`)**: 将计费思路具体化为详细的《计费/评级/考核标准》。
4.  **估算财务模型 (`表格`)**: 基于上述标准，进行详细的成本收益分析，产出《财务模型估算表》。

#### 步骤2：本体论、流程与系统细节设计
此步骤深入技术和流程细节，构建完整的系统蓝图。
1.  **设计核心对象 (`表格`+`文档`+`图示`)**:
    - **推导**: 依据《业务对象设计规范》，详细设计对象的**含义、关联关系、状态、属性、角色、类别**等。
    - **产出**: 《完整对象设计文档》和《对象关系图 (UML)》。
2.  **设计数据结构与权限 (`表格`+`图示`+`文档`)**:
    - **推导**: 基于对象及其属性，设计数据库表结构和字段（需含字段编号）；基于流程和节点，设计权限。
    - **产出**: 《字段说明清单》、ER图、《权限设计清单》。
3.  **设计流程与自动化 (`Excel`+`文档`+`图示`)**:
    - **推导**: 对象的“状态”变化定义了“流程”，流程的关键决策点定义了“审核节点”，流程中的重复性操作定义了“自动化”。
    - **产出**: 《流程与自动化设计总表》、详细的L3流程文档、包含数据流转和审核逻辑的《流程架构图》。
4.  **设计报表与看板 (`文档`+`表格`+`看板`)**:
    - **推导**: 对象的“属性”和“状态”决定了需要跟踪的数据和评估指标。
    - **产出**: 《指标清单》、《报表/看板设计文档》及部署逻辑。
5.  **设计人力对接流程 (`文档`)**:
    - **推导**: 基于《节点手册》和《L3流程文档》，为需要人工操作的节点设计标准化的培训、测试与认证流程。
    - **产出**: 《人力对接流程文档》，包含初次面试、入职培训、以及持续的节点测试标准。

## 3. 流程与自动化的逻辑关系

三类流程（数据、业务、报表）及其自动化是层层递进、相互依赖的，其核心推演逻辑如下：

-   **数据流程 (Data Process)**:
    -   **推演逻辑**: **由“对象”及其“属性”的CRUD（增删改查）需求决定。** 这是最底层的流程，是确保数据一致性和完整性的基础。
    -   **自动化逻辑**: 侧重于数据的**自动校验、清洗、备份和同步**，确保进入业务系统的数据是高质量的。

-   **业务流程 (Business Process)**:
    -   **推演逻辑**: **由“对象”的“状态”变迁驱动，由“节点”协作完成。** 例如，“素材”对象从“待审核”变为“已发布”，这个状态变化就定义了一个业务流程，其中可能包含“内容创作”、“内容审核”等多个步骤和节点。
    -   **自动化逻辑**: 侧重于**减少人工干预，实现流程的自动流转**。例如，一个节点完成操作后，任务自动流转到下一个审核节点；或者根据特定条件触发相应的业务操作。

-   **报表流程 (Reporting Process)**:
    -   **推演逻辑**: **由“业务流程”产生的数据以及管理决策的需求决定。** 业务流程运行产生大量过程数据，报表流程通过对这些数据的聚合、计算和分析，生成用于评估业务健康度和支持决策的指标与看板。
    -   **自动化逻辑**: 侧重于**数据的自动抽取、计算和呈现**。例如，每日定时从业务数据库中抽取数据，自动计算关键指标（KPIs），并更新到管理看板上。

总结：**对象定义数据流程 -> 对象的状态变迁定义业务流程 -> 业务流程的数据产出定义报表流程。** 自动化应优先保障数据质量，其次提升业务效率，最后实现决策支持的自动化。

## 4. 标准化执行流程

### 阶段一：顶层设计 (草稿)
1.  **梳理业务故事与核心要素 (`文档`+`表格`)**: 完成“阶段一：人为顶层设计”中的所有草稿输出。

### 阶段二：节点、财务与业务框架构建
1.  **固化节点与业务故事 (`表格`+ `文档`)**: 完成推演步骤1中的节点清单和业务故事。
2.  **设计计费与财务模型 (`文档`+`表格`)**: 完成推演步骤1中的计费标准和财务模型。

### 阶段三：本体论、流程与系统细节设计
1.  **设计对象、数据与权限 (`表格`+`文档`+`图示`)**: 完成推演步骤2中的对象、数据结构和权限设计。
2.  **设计流程、报表与自动化 (`Excel`+`文档`+`看板`)**: 完成推演步骤2中的流程、报表和自动化设计，并撰写L3文档。
3.  **设计人力对接流程 (`文档`)**: 基于节点和流程手册，设计标准化的培训与认证流程。

### 阶段四：部署、验证与迭代
1.  **系统与人力部署**: 部署线上`表格`+`看板`系统及自动化流程，并根据人力对接流程完成人员培训与上岗。
2.  **试运行与验证**: 通过人力流程试运行，并用实际数据验证财务模型的准确性。
3.  **跟踪与迭代**: 监控流程执行情况，持续迭代优化。

## 5. 核心产出物清单 (按推导顺序)

### 1. 顶层设计 (人为输入草稿)
- 业务故事草稿 (`文档`)
- 初始对象/节点/场域/流程清单 (`文档`)
- 计费思路 (`文档`)

### 2. 节点与业务框架 (推导产出)
- 完整业务故事 (`文档`)
- 分层节点清单与手册 (`表格`+`文档`)
- 计费/评级/考核标准 (`文档`+`表格`)
- 财务模型估算 (`表格`)

### 3. 本体论与系统细节 (推导产出)
- **对象相关**:
    - 完整对象设计文档 (含含义、状态、属性、角色、类别等) (`表格`+`文档`)
    - 对象关系图 (UML) (`图示`)
- **数据相关**:
    - 字段说明清单 (含含义、公式、**编号**) (`表格`)
    - 数据库ER图 (`图示`)
    - 非结构化文档模板与规范 (`文档`)
- **权限相关**:
    - 权限设计清单 (`表格`)
- **报表相关**:
    - 指标清单 (`表格/文档`)
    - 报表/看板设计文档 (`表格/文档`)
- **流程与自动化相关**:
    - 《流程与自动化设计总表》 (`Excel`)
    - L3流程文档 (`文档`)
    - 流程架构图 (含数据流转、审核节点) (`图示`)
- **人力资源相关**:
    - 人力对接流程文档 (含培训、面试、测试标准) (`文档`)

## 6. 项目管理与协作

### 6.1. 实施优先级
- **P0 (最高优)**: 业务故事 + 节点定义 + 初始对象 + 财务/薪酬定义。
- **P1 (次优)**: 详细对象设计 + 核心流程设计 + 表格系统搭建。
- **P2 (后续)**: 报表设计 + 自动化部署。

### 6.2. 团队协作模式
- **关系**: 后台组件团队负责提供标准化的模块设计（本SOP），前沿部署团队负责在具体业务中落地。
- **协同**: 细节由业务负责人完成，基础工具由 `ZH` 提供，系统验收由 `Silicon` 负责。

### 6.2.2. 调度中心代理
- **定义**: 对于某些高层级节点，其内部的任务排期和调度可以委托给一个统一的“调度中心”来代理执行，以提升效率和标准化水平。
- **应用**: 是否需要调度中心代理，应在《分层节点清单与手册》中进行明确标注。调度中心可以代理“任务分配”和“进度跟踪”的全部或部分。

### 6.3. 模块跟踪与Issue管理
模块的生命周期管理通过两份核心日志文档进行。

- **标准模块搭建日志**: 用于跟踪标准模块从0到1的开发过程，其核心内容包括：
    - **待办/完成日志**: 记录SOP各阶段产出物的完成情况。
    - **待议题清单**: 梳理设计和开发过程中需要讨论和明确的问题。
    - **POC验证记录**: 记录对核心**对象及其关键字段**进行的概念验证过程和结果。
    - **部署计划**: 明确业务流程和自动化的部署待办事项。

- **标准模块部署日志**: 以单个客户为单位进行管理，记录标准模块在实际业务中的落地过程，其关联流程如下：
    - **1. 提案会**: 向客户展示标准模块的价值和实施方案。
    - **2. Onboarding会议**: 按照《流程清单》和检查标准，对客户团队进行培训和交接。
    - **3. 定期跟踪**: 通过报表数据，持续跟踪流程执行效果。
    - **4. 质量监控**: 根据SOP中定义的检查标准，对核心对象的产出质量进行监控和反馈。

### 6.4. 代码与文档管理 (Github)
- **`/tool-packages`**: 存放通用工具包。
- **`/module-packages`**: 按模块存放（主文件、检查卡片、标准日志）。
- **`/service-logs`**: 存放具体项目的服务跟踪日志。
