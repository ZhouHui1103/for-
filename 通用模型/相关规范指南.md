# 薪酬文档搭建指南

1. 薪酬元规则/卡片分类：
    
    1. 我们会将薪酬元规则分为两个大类别，一个是效果类别，另外一个是动作类别
        
    2. 动作类别指：完整了某项具体的动作，付出了对应的时间和精力
        
    3. 效果类别指：A-某个项目/策略达到了某个节点/等级；B-对应结果（线索数/GMV/爆款）出现了xx个
        
2. 要素清单：
    
    1. 对应的，我们会分别有动作要素清单 + 效果要素清单
        
    2. **动作要素清单**：根据SOP本身结构来确定，应该尽可能完备刻画出需要花费时间的工作，但是要尽可能简洁。这类要素是只要按照工作流程+指南，付出对应时间，细心负责就可以完成达到，并且进入记录计算薪酬的。
        
    3. 动作要素例子：素材制作、内容策略打样、素材发布、单个需求包完成
        
    4. 动作要素含义（A）：我们一般会严格定义清晰动作要素的规范，例如素材发布是以发布+回传数据为准的，这样对应的可以严格把完整的动作细节包含进来，从而让薪酬计算和完成质量挂钩；少数情况下，我们会专门通过一个流程验收动作产出是否符合规范，只有符合规范才能算完成了动作。
        
    5. 动作要素含义（B）：一些很低频&碎片的工作，我们不会列入动作要素，例如助理需要注册/装修账号，我们不算入其内。对应的，这类动作的督导通过绩效评估来实现；这块的时间耗费一般也不大，可以整体被默认隐含地并入到其他要素的薪酬机制中。
        
    6. **效果要素清单**：根据SOP产出来确定，应该尽可能刻画出需要达到的最终业务结果。同时也尽可能简洁
        
    7. 效果要素例子：新助理入职（招募）、有效线索（引流）、
        
3. 如何设定薪酬规则：
    
    1. 薪酬维护责任人：最终熙康审核才可以生效，而日常中由BP来负责进行维护起草（包括数据核算）
        
    2. 动作-效果含义：优先用效果要素关联的薪酬卡片（刻画关键结果），而动作要素的薪酬卡片兜底（刻画基础动作）
        
    3. 工作属性偏好：成熟的确定性项目/工作（如铺量团），尽量用纯效果薪酬卡片；客观有不确定的项目，则用关键动作要素薪酬卡片，给负责工作的人提供安全感。
        
    4. 数字核算逻辑 - 动作要素薪酬卡片：
        
        1. 核心意义让对方付出时间是有基础回报安全感，但不宜高，因为最终要激励负责角色为效果负责。
            
        2. 本质是上做一个动作所需要的预期时间（一般取平均值） x 对应负责角色的时薪（取行业下限值）；
            
    5. 数字核算逻辑 - 效果要素薪酬卡片
        
        1. 核心意义是让对方能为结果负责，并且从中获得对应激励
            
        2. 对于相对有确定性的项目（一般为成熟项目），我们选择用效果要素薪酬卡片为主，甚至只用效果要素薪酬卡片
            
        3. 对应相对有一定不确定性的项目，我们选择用动作要素薪酬卡片兜底（但在合理范围内取下限），并用效果薪酬卡片作为主要激励
            
    6. 当效果要素变成一个相对确定性的事情时候，效果要素会实际趋向于动作要
        
        1. 效果要素本身也有了足够确定性，因此可以提供确定性
            
        2. 纯粹按照效果要素进行薪酬定义是更好的，因为此时可以倒过来筛选出有能力的人。（但是前提是任务不能太陡峭）｜此时整体模型也更简单？星星之石
            
        3. 效果要素的不确定性会影响整个模型会如何在两个极限情况间选择（动作要素兜底 vs 效果要素完全支撑）
            
4. 如何撰写薪酬元规则，一个薪酬元规则由以下部分组长：
    
    1. 编号：x-x，第一个数字和流程组相关，第二个数字则是流程组下具体的机制。（要素来自哪个流程组，目前具体数字暂时和流程编号不一样，但是是按流程组分类的）
        
    2. 概述：一句话描述薪酬元规则是什么含义
        
    3. 概念定义：一般同要素定义，一般情况下在这里严格定义相关要素即可
        
    4. 数据来源：注明是在哪里具体的数据表格中，可以看到对应的计算参数
        
    5. 相关注释：一段文字描述，一方面是让同事更容易理解规则本身，另一方面让同事理解其合理性
        
    6. 计算方法 & 参数：对应的元规则薪酬如何被计算，参数具体存在哪些
        
5. 如何检查薪酬元规则是否可以落地 - 和流程组文档结合
    
    1. 需要确保相应的要素是能在表格场域里被记录的，并且薪酬元规则也有标注是哪个场域
        
6. 相关核心文档：
    
    1. 薪酬要素清单：用于全局分析薪酬要素，以及相关具体分析
        
    2. 薪酬绩效文档：[薪酬绩效文档](https://x-silicon.feishu.cn/wiki/Gi4Qwn5Lpi157ikZ4UZcaNM1nIf)用于储存最后落地的规则
        
    3. 角色手册：在角色手册中的「」部分，我们会约定好
        

  

# 角色手册规范：

1. 内部角色 vs 外部角色
    
    1. **外部角色：**我们在招募时的基本单位，任何人成为一个新的外部角色（无论新人还是组织内同事），都需要严格的走一个双向选择流程，即我们按照相关流程考核对方，也尊重对方对角色本身的意愿
        
    2. **内部角色：**我们进行业务分配&薪酬机制设定时候，是以内部角色为单位的。只有一个伙伴已成为某个外部角色，我们要在这个范围内进行内部角色划分，除了薪酬机制需要重新确认外，我们是无需再进行双向选择流程，我们可以直接安排相关调动。
        
    3. **角色手册是以单个外部角色来建立文档的**，但是文档内也会详细说明内部角色划分和相关内容。
        
    4. 外部角色是一揽子的内部角色的组合，这样方便管理。例如做了HRA，基本上就肯定愿意&够格做HRB，因此我们在人力管理上就统一为HR这样一个外部角色
        
    5. 针对同一个外部角色和对应的工作范围，在内部进行业务分配和薪酬激励等相关时，肯定存在更细分的划分，因此我们就会约定进一步的细分角色。
        
    6. 一个伙伴加入我们，是可以同时做多个外部角色/内部角色的，角色是被挂载在“人”上的。
        
2. 为什么需要角色手册：
    
    1. 用于完整定义一个角色的各方面核心内容：基础内容 + 生命周期旅途内容 + 其他
        
    2. 不同流程都会调用角色手册的相关信息，例如招募流程会用到具体某个角色手册中的考核流程设定
        
    3. 方便相关同事通过角色手册快速理解自身定位&相关工作内容
        
3. 角色手册应包含的信息：
    
    1. 角色基础信息：外部角色概述 + 内部细分角色 + 细分角色概述
        
    2. 生命周期板块（A）：相关板块的具体设定 + 详细流程&机制 + 引用相关文档
        
    3. 生命周期：招募 - 培训 - 日常 - 薪酬绩效
        
4. 信息的「精确储存 vs 简要说明」
    
    1. 简要说明信息：部分核心信息是专门存放在通用文档里的，例如薪酬机制在[薪酬绩效文档](https://x-silicon.feishu.cn/wiki/Gi4Qwn5Lpi157ikZ4UZcaNM1nIf)中。他们都有专门的文档并被详细维护，因此我们不会在角色手册里再重复详细描述。
        
    2. 简要说明信息：也许原始位置的文档的细节信息一直会变化，但是总体逻辑不会变，因此我们会在角色手册里主要说明核心逻辑。例如薪酬部分我们不会直接说明具体规则，但是会说明核心逻辑。**简要信息的目的是方便相关人理解对应原始文档里的精准信息。**
        
    3. 精确储存信息：需要在角色手册里被清晰约定的信息，且角色手册就是这类信息的第一来源。
        
    4. 简要说明信息：薪酬绩效机制、培训机制、日常工作内容、需后续掌握技能；
        
    5. 精确储存信息：角色基础信息、考核相关设定，相关工作文档目录

# 业务对象设计规范

## 整体哲学：

1. 业务对象设计是所有流程设计的起点和最终收尾点
    
2. 业务对象客观存在，我们需要从实际业务模型中去做抽离抽象
    
3. 我们会按照模块/层次，将关联比较紧密，或者同属一个大概念的业务对象放到同一层
    
4. 业务对象决定了：
    
    1. 应当有哪些流程：每个有状态属性的变量应当都有
        
    2. 如何评估业务质量：首先是如何评估对象本身，随后是哪些角色该背负这些指标
        
    3. 如何定义薪酬规则：业务对象往往对应了动作和效果要素，且最终薪酬结算的数据来源一定是对象数据
        
    4. 我们会需要使用怎样的数据库/数据报表/工作文档：这些都是业务对象的数据化表达的衍生
        

  

## 对象设计规范

1. 如果要描述一个对象，我们有这几个核心部分
    
    1. 概述：一句话描述对象的精准定义
        
    2. 属性：对象存在哪些属性，其中部分属性会自动推演出对象应当存在哪些字段和子对象
        
    3. 状态：对象存在哪些状态，这里可以衍生很多东西。推演流程、任务对象等都与之相关
        
    4. 类别：提前设计好对象有哪些类别，很重要，有时候可以传递很多关键信息
        
    5. 角色关系：填写这个数据（维护角色）、负责做动作产生数据（执行角色）、对数据指标负责（责任角色）、审核角色（含审核流）
        
    6. 对象关系：和其他对象是否存在某种特定关系
        
    7. 关联字段：这个对象和数据库中哪些其他对象是关联的。记得注明单选/多选，以及是否有筛选
        
    8. 其他字段：其他的关系型字段，有时候我们也会进一步分类去罗列。（时间相关、负责人相关等）
        
    9. 文档字段：即非关系形字段，这里需要说明清晰
        
    10. 储存场域：这里明确分为表格场域 + 文档场域两种
        
    11. 关联报表：这个对象的数据最后会流入到哪些报表中（不一定是原始数据，但总归说明报表在关心这个对象）
        
    12. 公式注明：如果某个字段是可以被公式直接计算的，在这里注明。并简要说明计算逻辑，或者严格标注。
        
    13. 审核说明：说明下这个角色审核流程
        
    14. 评估指标（短期｜目标属性）：如何评估时间长度｜或者
        
    15. 分配额度（任务属性）：特定角色在周期时间内可以承接多少个这样任务
        
    16. 关联流程：哪个L3流程在驱动这个状态变量
        
2. 属性板块：每个属性都内置了一部分字段和对应逻辑
    
    1. 模块接口：如果某一层变量主要靠这个变量和其他模块互动的话，就标注这个模块
        
    2. 短期 vs 长续：
        
        1. 短期变量一般我们是要衡量这个变量状态演变的时间的，速度默认会成为该对象的一个评估标准
            
        2. 对应的，短期变量一定存在：排期完成时间、提出时间、
            
        3. 长续变量一般是会稳定存在的，它一般是用来被其他变量挂载的根
            
    3. 状态变量 vs 被动承接：
        
        1. 状态变量是指本身存在状态变化的变量，一旦存在状态变化就意味着它会有对应L3流程来支持
            
        2. 而被动承接指其只是用来储存数据的数据对象，不会有专门流程。他们一般都是被挂载到状态变量上作为状态变量的一种数据记录补充
            
    4. 目标属性
        
        1. 一旦一个变量存在目标属性，那么这个变量一定会有一个评价指标，这个指标可以是某个率，或者等级
            
        2. 我们便会在「评估指标」这里罗列相关指标+具体的严格定义
            
        3. 部分情况下，评估指标会和对象的衍生对象的评估指标相关
            
    5. 指令属性：部分字段是指令方用于下指令的字段，我们在「其他字段」里可以单独描述
        
    6. 任务类型：说明这个对象是个任务，而任务则是源自某个状态对象状态迁移的需求（我们称原始对象）。一般来说一个任务即一次状态变迁。
        
        1. 字段一定有：原始对象（关联）、时间相关（提出时间、排期时间，实际验收时间）、人员相关（提出人、负责人、督导人、执行人）
            
        2. 类别字段一定有：原始对象的状态，或者状态迁移动作
            
        3. 本身也是个状态变量，且状态为：已提出、已排期、执行中、已验收
            
        4. 对应的也一定下游会有督导对象，来督导任务完成情况
            
    7. 定期触发：表明应当存在一个流程，定期流程触发并且产生这个对象的一行记录。
        
    8. 督导属性：表明这个对象是专门用来承接督导数据的 - 一般和督导行为挂钩
        
    9. 版本管理：表明这个对象是有版本属性的，会存在同一行记录版本更新。
        
    10. 二次数据
        
3. 对象间关系说明：有时候我们需要单独附一个说明文档，讲解对象间关系。
